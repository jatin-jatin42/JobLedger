SECTION 2: SYSTEM OVERVIEW
================================================================================

A single-user-per-account job application tracker. Each registered user manages
only their own applications. There is no admin role.

Core entities:
  - User          (authentication + ownership)
  - Application   (a job they applied to)
  - Note          (timestamped notes attached to an application)

The most critical business rule is the APPLICATION STATE MACHINE.
Applications move through stages in a defined, enforced order.
No stage can be skipped or reversed (except explicit rejection from any stage).

================================================================================
SECTION 3: APPLICATION STATE MACHINE (CRITICAL — ENFORCE THIS STRICTLY)
================================================================================

Valid Stages (in order):
  1. APPLIED
  2. SCREENING
  3. INTERVIEW
  4. OFFER
  5. REJECTED  (terminal — reachable from ANY stage)
  6. WITHDRAWN (terminal — reachable from ANY stage, user withdrew themselves)

Transition Rules:
  - APPLIED      → SCREENING, REJECTED, WITHDRAWN
  - SCREENING    → INTERVIEW, REJECTED, WITHDRAWN
  - INTERVIEW    → OFFER, REJECTED, WITHDRAWN
  - OFFER        → REJECTED, WITHDRAWN (edge case: offer rescinded)
  - REJECTED     → (no further transitions — terminal)
  - WITHDRAWN    → (no further transitions — terminal)

Implementation requirement:
  - Define these transitions as a constant dictionary/map in the backend.
  - The stage update endpoint MUST validate against this map.
  - Return HTTP 422 with a descriptive error if an invalid transition is attempted.
  - Never trust the frontend to enforce this — backend is the single source of truth.

Example enforcement (Python pseudocode):
  VALID_TRANSITIONS = {
      "APPLIED":    ["SCREENING", "REJECTED", "WITHDRAWN"],
      "SCREENING":  ["INTERVIEW", "REJECTED", "WITHDRAWN"],
      "INTERVIEW":  ["OFFER", "REJECTED", "WITHDRAWN"],
      "OFFER":      ["REJECTED", "WITHDRAWN"],
      "REJECTED":   [],
      "WITHDRAWN":  [],
  }

SECTION 4: DATABASE SCHEMA
================================================================================

Table: users
  - id            UUID, primary key, default gen_random_uuid()
  - email         VARCHAR(255), unique, not null
  - password_hash VARCHAR(255), not null
  - full_name     VARCHAR(100), not null
  - created_at    TIMESTAMP, default now()

Table: applications
  - id            UUID, primary key
  - user_id       UUID, foreign key → users.id, on delete CASCADE
  - company_name  VARCHAR(150), not null
  - role_title    VARCHAR(150), not null
  - job_url       TEXT, nullable
  - location      VARCHAR(150), nullable
  - salary_min    INTEGER, nullable (in local currency, no decimals)
  - salary_max    INTEGER, nullable
  - stage         VARCHAR(20), not null, default 'APPLIED'
                  CHECK stage IN ('APPLIED','SCREENING','INTERVIEW','OFFER','REJECTED','WITHDRAWN')
  - applied_date  DATE, not null, default today
  - created_at    TIMESTAMP, default now()
  - updated_at    TIMESTAMP, default now(), auto-update on change

Table: notes
  - id            UUID, primary key
  - application_id UUID, foreign key → applications.id, on delete CASCADE
  - user_id       UUID, foreign key → users.id, on delete CASCADE
  - content       TEXT, not null
  - created_at    TIMESTAMP, default now()

Indexes:
  - applications(user_id)
  - applications(user_id, stage)
  - notes(application_id)

Constraints:
  - salary_min must be <= salary_max if both provided (enforce in app layer)
  - content of notes must not be empty/whitespace only
  - applied_date cannot be in the future

SECTION 5: BACKEND API SPECIFICATION
================================================================================

All endpoints return JSON.
All protected endpoints require: Authorization: Bearer <access_token>
All error responses follow this shape:
  { "error": "message describing the problem" }

All success responses for lists:
  { "data": [...], "total": N }

All success responses for single items:
  { "data": { ... } }

--------------------------------------
AUTH ENDPOINTS (no auth required)
--------------------------------------

POST /api/auth/register
  Request:  { email, password, full_name }
  Response: 201 { data: { id, email, full_name, created_at } }
  Errors:
    - 400 if any field missing or invalid format
    - 409 if email already registered
  Validation:
    - email must be valid format
    - password minimum 8 characters
    - full_name minimum 2 characters, no special chars

POST /api/auth/login
  Request:  { email, password }
  Response: 200 { data: { access_token, user: { id, email, full_name } } }
  Errors:
    - 401 if credentials invalid (use same message for both — no enumeration)

GET /api/auth/me
  Auth required.
  Response: 200 { data: { id, email, full_name, created_at } }

--------------------------------------
APPLICATION ENDPOINTS (auth required)
--------------------------------------

GET /api/applications
  Query params:
    - stage (optional, filter by stage)
    - search (optional, matches company_name or role_title, case-insensitive)
    - sort_by (optional: applied_date | updated_at | company_name, default: updated_at)
    - order (optional: asc | desc, default: desc)
  Response: 200 { data: [...applications], total: N }
  Rules:
    - Only return applications belonging to the authenticated user.

POST /api/applications
  Request:
    {
      company_name,   required
      role_title,     required
      job_url,        optional
      location,       optional
      salary_min,     optional integer
      salary_max,     optional integer
      applied_date    optional, ISO date string, defaults to today
    }
  Response: 201 { data: { ...application } }
  Errors:
    - 400 on missing required fields or invalid data
    - 422 on business rule violation (salary_min > salary_max, future applied_date)

GET /api/applications/:id
  Response: 200 { data: { ...application, notes: [...] } }
  Errors:
    - 404 if not found or not owned by user

PATCH /api/applications/:id
  Allowed fields to update:
    company_name, role_title, job_url, location, salary_min, salary_max, applied_date
  NOTE: stage is NOT updated through this endpoint.
  Response: 200 { data: { ...application } }
  Errors:
    - 400 on invalid data
    - 404 if not found or not owned by user

PATCH /api/applications/:id/stage
  Request:  { stage: "SCREENING" }
  Response: 200 { data: { ...application } }
  Errors:
    - 422 if transition is invalid (include message listing valid next stages)
    - 404 if not found or not owned by user

DELETE /api/applications/:id
  Response: 204 No Content
  Errors:
    - 404 if not found or not owned by user

--------------------------------------
NOTES ENDPOINTS (auth required)
--------------------------------------

GET /api/applications/:id/notes
  Response: 200 { data: [...notes], total: N }
  Errors:
    - 404 if application not found or not owned by user

POST /api/applications/:id/notes
  Request:  { content }
  Response: 201 { data: { id, application_id, content, created_at } }
  Errors:
    - 400 if content is empty or whitespace only
    - 404 if application not found or not owned by user

DELETE /api/applications/:id/notes/:note_id
  Response: 204 No Content
  Errors:
    - 404 if note or application not found or not owned by user

--------------------------------------
DASHBOARD ENDPOINT (auth required)
--------------------------------------

GET /api/dashboard
  Response:
    {
      "data": {
        "total": N,
        "by_stage": {
          "APPLIED": N,
          "SCREENING": N,
          "INTERVIEW": N,
          "OFFER": N,
          "REJECTED": N,
          "WITHDRAWN": N
        },
        "recent_activity": [ last 5 updated applications ]
      }
    }

SECTION 10: CLAUDE.MD (AI GUIDANCE FILE)

Create a file called CLAUDE.md in the project root with the following content:

---
# CLAUDE.md — AI Agent Guidance for Job Application Tracker

## Purpose
This file defines rules and constraints for AI agents working on this codebase.
These rules protect system integrity and must never be violated.

## Non-Negotiable Rules

### State Machine
- The VALID_TRANSITIONS map in application_service.py is the single source of truth.
- Never add a new stage without updating VALID_TRANSITIONS.
- Never bypass transition validation — not even in tests.
- Stage changes must ONLY go through the /stage endpoint, never through PATCH /applications/:id.

### Ownership
- Every data-fetching query on applications and notes MUST filter by user_id.
- Never return data without confirming it belongs to the authenticated user.
- Ownership check must live in the service layer, not just the route.

### Validation
- All input validation must use Marshmallow schemas.
- Never manually parse or trust raw request data without schema validation.
- Schema files are in app/schemas/ — do not inline validation logic in routes.

### Tests
- Every new endpoint must have at least one corresponding test.
- Stage transition tests must be updated when transitions change.
- Never delete existing tests — fix them instead.

### No Clever Code
- Prefer readable, explicit code over clever one-liners.
- No dynamic attribute access (getattr with string) unless unavoidable.
- No global state outside of Flask extensions.

## Conventions
- Routes only handle HTTP concerns (parse input, call service, return response).
- Services handle all business logic.
- Models are plain SQLAlchemy models — no business logic in models.
- UUIDs for all primary keys — no integer IDs.
- All datetimes stored as UTC.

## Adding Features
Before adding a feature:
1. Identify which service file it belongs in.
2. Check if a new schema is needed.
3. Write the test first (TDD preferred).
4. Do not modify the state machine without explicit instruction.

## What Not To Do
- Do not add new libraries without checking if existing ones cover the need.
- Do not expose stack traces in API error responses.
- Do not store plaintext passwords anywhere, ever.
- Do not add CORS wildcard (*) in production config.

SECTION 11: ERROR HANDLING STANDARDS
================================================================================

- Never return raw exception messages to the client.
- Use a centralized error handler registered in create_app().
- HTTP status codes must be semantically correct:
    200 OK, 201 Created, 204 No Content
    400 Bad Request (invalid input format)
    401 Unauthorized (missing or invalid token)
    403 Forbidden (authenticated but not allowed)
    404 Not Found (resource doesn't exist or not owned)
    409 Conflict (duplicate email)
    422 Unprocessable Entity (business rule violation)
    500 Internal Server Error (unexpected — log it, return generic message)

- Log all 5xx errors with enough context to diagnose (but never log passwords).
- Marshmallow validation errors → 400 with field-level error messages.
- JWT errors → 401 with message "Authentication required."
